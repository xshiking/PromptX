## Context

### 背景

当前 PromptX 的角色知识主要以 Markdown 形式存在，并在角色激活（`action`）阶段被解析/展开后注入上下文。随着知识规模增长，这种“全量注入”会带来：

- 上下文长度膨胀与噪音上升（无关知识被迫进入每次对话）
- 无法随问题动态挑选最相关知识片段（命中不稳定、幻觉风险增加）

本变更引入**独立于记忆网**（`cognition`）的“角色知识库”系统，使用 SQLite FTS5 建立全文索引，并在桌面端对用户无感地进行**自动检索注入**：每次用户提问前检索 TopK 片段并注入到提示词中。

### 当前状态（相关组件）

- **资源体系**：通过 `ResourceManager.loadResource("@role://...")`、`@knowledge://...` 等协议加载 Markdown 内容。
- **记忆体系**：通过 `CognitionManager/Memory/Network` 管理 `~/.promptx/cognition/<roleId>/` 的记忆网络（本变更不复用、不污染）。
- **MCP**：`packages/mcp-server` 通过 `cli.execute(...)` 暴露工具（如 `action/recall/remember`）。
- **桌面端**：负责最终“发送给模型的 prompt 组装”，适合承载“对用户无感”的自动注入。

### 约束与假设

- 运行环境允许使用 `better-sqlite3`；SQLite 构建需支持 **FTS5**。需要在初始化阶段做能力探测，并提供清晰报错与降级策略。
- 知识来源优先来自 role 文档 `<knowledge>` 内的 `@knowledge://...` 引用清单（稳定可控），避免在 v0 进行大范围目录扫描带来不确定性。
- 自动注入必须有**长度预算**，并能在失败时“静默降级”（不影响对话主流程）。

## Goals / Non-Goals

**Goals:**

- 为每个 `roleId` 提供独立的知识索引库：`~/.promptx/knowledge/<roleId>/knowledge.db`。
- 支持知识源文档的**增量索引**（基于内容 hash 识别变更）。
- 提供可复用的检索 API，并通过 Core CLI + MCP 工具对外提供：`knowledge.search` / `promptx_knowledge_search`。
- 桌面端每轮对话自动检索并注入 TopK 片段，做到**对用户无感**。
- 检索结果必须可解释：包含 `source_path`（资源 id）与 `title_path`（标题路径），便于追溯与引用。
- 与记忆网完全解耦：不读写 `~/.promptx/cognition/<roleId>/`，不影响 `recall/remember` 行为。

**Non-Goals:**

- 不引入向量检索（embedding）或外部向量数据库。
- 不在 v0 实现跨角色混合检索与重排（按 `roleId` 隔离）。
- 不在 v0 实现“知识写回/学习”闭环（知识库先只读；编辑与发布走既有资源流程）。
- 不在 v0 追求完美中文分词；FTS5 的 `unicode61` 先满足可用性，后续再增强。

## Decisions

### 1) 选择 SQLite FTS5 作为检索引擎

**决策**：使用 SQLite FTS5 + BM25 排序做全文检索。

**理由**：

- 依赖少、离线可用、部署简单（与现有 `better-sqlite3` 生态一致）
- 查询速度快，适合“每轮对话前自动检索”
- 支持 snippet 生成，便于注入时控制长度

**备选方案**：

- 自建倒排索引（JSON）：实现简单但排序与性能较弱，且维护复杂度更高。
- 向量检索：体验更好但引入依赖、数据迁移与成本，非 v0 目标。

### 2) 知识库与记忆网严格隔离

**决策**：知识库不写入 `cognition`（`engrams.db/network.json`），不复用 cue 网络。

**理由**：

- 知识是稳定可发布资产，记忆是用户侧经验网络，两者生命周期/治理不同
- 避免知识污染记忆召回域，导致 recall 结果失真
- 便于为知识库设置单独的注入策略与预算

### 3) 以“资源 id”为 source_path 的统一标识

**决策**：索引记录 `source_path` 使用 `@knowledge://...` 这类资源 id，而非磁盘绝对路径。

**理由**：

- 跨设备/跨安装路径一致
- 与现有 `ResourceManager` 协议体系对齐
- 便于在 UI/调试中直接复用资源加载能力进行回溯

### 4) 知识来源的确定：优先依赖 role 的引用清单

**决策**：v0 索引来源只取 `@role://<roleId>` 的 `<knowledge>` 中解析出的 `@knowledge://...` 引用清单（可选后续再加目录扫描兜底）。

**理由**：

- 可控、显式、避免扫描带来的误收录
- 与当前 role 语义结构一致（DPML 语义占位符）

### 5) 自动注入放在桌面端（宿主侧）实现

**决策**：对用户无感的“每轮自动检索注入”由桌面端在发送给模型前完成，调用 MCP `promptx_knowledge_search` 获取 TopK 并注入。

**理由**：

- Core/CLI 不掌控“每轮 prompt 组装”时机；宿主侧是天然挂载点
- 避免依赖模型是否想起调用工具，提升稳定性
- 宿主侧更易统一做预算控制/降级策略/可视化调试

### 6) 文档切片策略：按标题层级 + 段落累积

**决策**：Markdown 解析采用轻量切片：标题（#、##、###）作为边界，内容按段落累积并限制 chunk 尺寸。

**理由**：

- 兼容性好，能较好保留语义结构
- 易于构建 `title_path`，增强可解释性
- 实现成本低、适合 v0

**备选**：

- AST 级 Markdown 解析：更准确但增加依赖与复杂度（后续再评估）。

## Risks / Trade-offs

- **[FTS5 不可用]** → 初始化阶段探测（尝试创建 FTS5 虚表）；失败时给出明确错误，并提供降级（如提示用户/禁用自动注入/回退到 LIKE 检索）。
- **[中文检索质量一般]** → v0 接受；后续可加 query 扩展、同义词表、或引入更强 tokenizer/embedding。
- **[自动注入导致上下文过长]** → 设定注入预算（TopK、每条 snippet 长度、总字符上限）；必要时仅注入 snippet，不注入全 content。
- **[索引更新成本]** → 采用 sources hash 增量更新；仅在内容变化时重建该 source 的 docs。
- **[噪音注入/误命中]** → 采用稳定输出格式与来源标注；必要时在宿主侧添加阈值（低相关度不注入）与调试开关。
